# AGENTS.md

Operational guide for AI coding agents. Read this **before** making changes.

## 1. What this repo is

`gh-sherpa` is a [GitHub CLI](https://github.com/cli/cli) extension written in Go that creates branches and pull requests linked to GitHub or Jira issues, following a configurable contribution model.

Entrypoint: [`main.go`](main.go) → [`cmd/root.go`](cmd/root.go) (cobra).

## 2. Repo layout

```
cmd/                        Cobra commands (presentation layer)
  root.go                   Root command, registers subcommands and persistent flags
  create_branch/            `sherpa create-branch` (alias `cb`)
  create_pull_request/      `sherpa create-pr`     (alias `cpr`)
  common/                   Shared command helpers (fork setup, etc.)

internal/
  use_cases/                Business logic — CreateBranch, CreatePullRequest
  domain/                   Interfaces consumed by use cases (mock targets)
  branches/                 Branch naming + type resolution
  config/                   Config loading + default-config.yml
  gh/                       GitHub CLI provider (wraps `gh`)
  git/                      Git provider (wraps `git`)
  interactive/              Survey-based prompts
  issue_trackers/           GitHub + Jira issue tracker providers
  fork/                     Fork detection and setup
  logging/                  User-facing output (use this, not fmt.Println)
  mocks/                    AUTOGENERATED by mockery — do not edit by hand
  fakes/                    Hand-written test doubles
  metadata/, validator/, utils/

pkg/                        Public packages (limited)

docs/
  USAGE.md                  Source of truth for CLI flags
  ARCHITECTURE.md           Use case flow diagrams
```

## 3. Architecture rules

The project follows a lightweight hexagonal pattern:

```
cmd/  ──►  internal/use_cases/  ──►  providers (gh, git, issue_trackers, branches, interactive)
```

- **`cmd/` must not call `gh`, `git`, or HTTP APIs directly.** It only parses flags and wires up a use case.
- **`internal/use_cases/` must not shell out directly either.** It depends on interfaces from `internal/domain/`.
- Provider implementations live in their own package and satisfy the interfaces in `internal/domain/`.
- Mocks are generated from `internal/domain/` per [`.mockery.yaml`](.mockery.yaml). Run `make generate-mocks` after changing any domain interface.

## 4. Build, test, verify

Go 1.21 (see [`.tool-versions`](.tool-versions)). Make targets:

| Command | What it does |
|---|---|
| `make build` | `go build -o bin/ ./...` |
| `make test` | `go test ./...` |
| `make coverage` | Coverage profile in `.local/coverage.out` |
| `make generate-mocks` | Regenerate mocks via mockery v2.32.4 |
| `make verify` | tidy + checkfmt + lint + vet + build + test + coverage — **required before opening a PR** |

## 5. Coding conventions

- **Errors**: return plain errors up the stack; the CLI prints them through `internal/logging`. Avoid wrapping just to wrap.
- **User output**: use `internal/logging` (`logging.Print*`, `logging.Error*`, `logging.Debug*`) — not `fmt.Println`.
- **Interactivity**: any new prompt must respect both non-interactive switches:
  - `--yes` / `-y` → use defaults
  - `--output json` → implies non-interactive (no stdin reads, JSON only on stdout)

  Existing pattern: [`cmd/create_branch/cmd.go:84-88`](cmd/create_branch/cmd.go).
- **JSON output**: if you add fields to `--output json`, update the documented shape in [`docs/USAGE.md`](docs/USAGE.md). Current shapes:
  - `create-branch`: `{"branch":"<name>"}`
  - `create-pr`: `{"branch":"<name>","pr_url":"<url>","draft":<bool>}`
- **Errors in JSON mode**: emit `{"error":"..."}` on stderr and exit non-zero (see existing pattern in both commands).
- **Branch / PR types**: defined in [`internal/config/default-config.yml`](internal/config/default-config.yml). Do not hard-code prefixes; resolve through `branches.Provider`.

## 6. How to add a new command

1. Create `cmd/<name>/cmd.go` mirroring [`cmd/create_branch/cmd.go`](cmd/create_branch/cmd.go) (struct `<name>Flags`, `init()`, `runCommand`, `preRunCommand`).
2. Register in [`cmd/root.go`](cmd/root.go) with `rootCmd.AddCommand(...)`.
3. Put business logic in `internal/use_cases/<name>.go` and declare its dependencies as interfaces in `internal/domain/`.
4. Run `make generate-mocks`, add tests using the generated mocks.
5. Document the command and all its flags in [`docs/USAGE.md`](docs/USAGE.md).

## 7. How to add a new flag

- Add it to the flag struct and bind it in `init()` using the existing `PersistentFlags()` pattern.
- Decide non-interactive behavior up front: what should happen with `--yes` and `--output json`?
- Document it in [`docs/USAGE.md`](docs/USAGE.md) — including short form if any. **The flag list in USAGE.md is the source of truth**, not the README.
- If the flag is AI-workflow relevant, mention it in the README's "AI-assisted development" section too.

## 9. Issue trackers

Detection happens in [`internal/issue_trackers/parser.go`](internal/issue_trackers/parser.go):

- Numeric (`123`) → GitHub
- `PROJECTKEY-123` → Jira (requires Jira credentials in `~/.config/sherpa/config.yml`; first run prompts for them in interactive mode)

When adding new tracker support, follow the existing `internal/issue_trackers/<tracker>.go` pattern and register it in `NewFromConfiguration`.

## 10. Pre-PR checklist for agents

- [ ] `make verify` passes locally
- [ ] If you changed any interface in `internal/domain/`, you ran `make generate-mocks`
- [ ] If you added or changed a CLI flag, `docs/USAGE.md` is updated
- [ ] If you added or changed `--output json` shape, the documented shape in `docs/USAGE.md` matches
- [ ] No new interactive prompts that ignore `--yes` or `--output json`
- [ ] Branch name follows the sherpa convention (`feature/`, `bugfix/`, `hotfix/`, `refactoring/`, …)
- [ ] PR title and body reference the issue being addressed

## 11. Do not touch without a strong reason

- `internal/mocks/` — regenerated by mockery, never edit by hand
- `internal/config/default-config.yml` — changes default behavior for every user
- `.drafterconfig.yml`, `repolinter.json`, `sonar-project.properties` — CI / quality tooling
- `CHANGELOG.md` — drafted automatically
- `go.sum`, `gh-sherpa` binary, `version` file
